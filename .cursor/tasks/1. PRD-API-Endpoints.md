# Product Requirements Document: Core API Endpoints

## 1. Problem Statement

The AI Interview Assistant backend currently has a basic health check endpoint and a rubric creation endpoint, but lacks the core API endpoints needed for the full interview workflow. Teachers need to be able to:

- List and retrieve rubrics they've created
- Create interview templates (interview plans) that define the structure and questions for interviews
- Create new interviews based on templates
- Retrieve interview details to view progress and results

Without these endpoints, the frontend cannot complete the core user journey of creating rubrics, setting up interview templates, and conducting interviews with students.

## 2. Target Users

- **Primary Users**: Teachers who use the AI Interview Assistant to conduct diagnostic or formative assessment interviews with students
- **Secondary Users**: System administrators who may need to manage or audit interview data

## 3. User Stories

### 3.1 Rubric Management
- **As a teacher**, I want to view all my rubrics in a list, so I can select one when creating an interview template.
- **As a teacher**, I want to see the details of a specific rubric, so I can verify its contents before using it.

### 3.2 Interview Template Creation
- **As a teacher**, I want to create an interview template linked to a rubric, so I can define the structure and questions for conducting interviews.
- **As a teacher**, I want to specify curriculum alignment information (subject, level band) when creating a template, so I can organize and filter templates later.

### 3.3 Interview Management
- **As a teacher**, I want to create a new interview from a template, so I can start conducting an interview with a student.
- **As a teacher**, I want to retrieve interview details by ID, so I can view the interview status, messages, and results.

## 4. Functional Requirements

### 4.1 POST /rubrics
**Status**: ✅ Already implemented (handler exists in `handlers/rubrics.go`)

**Purpose**: Create a new rubric for a teacher.

**Request Body**:
```json
{
  "teacherId": "uuid",
  "title": "string (required)",
  "description": "string (optional)",
  "rawText": "string (required)"
}
```

**Response**: 201 Created
```json
{
  "rubricId": "uuid",
  "teacherId": "uuid",
  "title": "string",
  "description": "string",
  "rawText": "string",
  "isEnabled": true,
  "createdAt": "timestamp",
  "updatedAt": "timestamp"
}
```

**Validation**:
- `title` must not be empty
- `teacherId` must be a valid UUID
- `rawText` must not be empty

### 4.2 GET /rubrics
**Status**: ⚠️ Needs implementation

**Purpose**: List all rubrics for a teacher.

**Query Parameters**:
- `teacherId` (required): UUID of the teacher

**Response**: 200 OK
```json
[
  {
    "rubricId": "uuid",
    "teacherId": "uuid",
    "title": "string",
    "description": "string",
    "rawText": "string",
    "isEnabled": true,
    "createdAt": "timestamp",
    "updatedAt": "timestamp"
  }
]
```

**Error Responses**:
- 400 Bad Request: Missing or invalid `teacherId` query parameter
- 500 Internal Server Error: Database error

**Implementation Notes**:
- Use `ListRubricsByTeacher` query from `db.Queries`
- Return empty array if teacher has no rubrics
- Order by `created_at DESC` (newest first)

### 4.3 POST /interview-templates
**Status**: ⚠️ Needs implementation

**Purpose**: Create a new interview template (interview plan) linked to a rubric.

**Request Body**:
```json
{
  "rubricId": "uuid (required)",
  "title": "string (required)",
  "instructions": "string (optional)",
  "config": {} (optional, JSON object),
  "status": "draft" | "in_progress" | "completed" (optional, defaults to "draft"),
  "curriculumSubject": "string (optional)",
  "curriculumLevelBand": "string (optional)"
}
```

**Response**: 201 Created
```json
{
  "interviewPlanId": "uuid",
  "rubricId": "uuid",
  "title": "string",
  "instructions": "string",
  "config": {},
  "status": "draft",
  "curriculumSubject": "string",
  "curriculumLevelBand": "string",
  "createdAt": "timestamp",
  "updatedAt": "timestamp"
}
```

**Validation**:
- `rubricId` must be a valid UUID and reference an existing rubric
- `title` must not be empty
- `status` must be one of: "draft", "in_progress", "completed" (defaults to "draft")
- `config` must be valid JSON (if provided)

**Error Responses**:
- 400 Bad Request: Invalid request body, missing required fields, or invalid rubric reference
- 404 Not Found: Referenced rubric does not exist
- 500 Internal Server Error: Database error

**Implementation Notes**:
- Use `CreateInterviewPlan` query from `db.Queries`
- Verify rubric exists before creating plan (use `GetRubricByID`)
- Default `config` to empty JSON object `{}` if not provided
- Default `status` to "draft" if not provided

### 4.4 POST /interviews
**Status**: ⚠️ Needs implementation

**Purpose**: Create a new interview instance from an interview template.

**Request Body**:
```json
{
  "interviewPlanId": "uuid (required)",
  "teacherId": "uuid (optional)",
  "simulated": true (optional, defaults to true),
  "studentName": "string (optional)",
  "status": "draft" | "in_progress" | "completed" (optional, defaults to "in_progress")
}
```

**Response**: 201 Created
```json
{
  "interviewId": "uuid",
  "interviewPlanId": "uuid",
  "teacherId": "uuid",
  "simulated": true,
  "studentName": "string",
  "status": "in_progress",
  "startedAt": "timestamp",
  "completedAt": null
}
```

**Validation**:
- `interviewPlanId` must be a valid UUID and reference an existing interview plan
- `status` must be one of: "draft", "in_progress", "completed" (defaults to "in_progress")
- `simulated` defaults to `true` if not provided

**Error Responses**:
- 400 Bad Request: Invalid request body, missing required fields, or invalid interview plan reference
- 404 Not Found: Referenced interview plan does not exist
- 500 Internal Server Error: Database error

**Implementation Notes**:
- Use `CreateInterview` query from `db.Queries`
- Verify interview plan exists before creating interview (use `GetInterviewPlanByID`)
- Default `status` to "in_progress" if not provided
- Default `simulated` to `true` if not provided
- `startedAt` is automatically set by database

## 5. Non-Goals

The following features are explicitly **not** included in this PRD:

- **Authentication/Authorization**: No user authentication or authorization checks (assumes this will be handled by middleware later)
- **Pagination**: List endpoints do not include pagination (can be added in future iteration)
- **Filtering/Sorting**: GET /rubrics does not support filtering by enabled status or sorting options
- **Interview Updates**: No PATCH/PUT endpoints for updating interviews (only creation and retrieval)
- **Interview Messages**: No endpoints for retrieving or managing interview messages (separate feature)
- **Interview Summaries**: No endpoints for interview summaries or criterion evidence (separate feature)
- **Nested Resources**: GET /interviews/{id} does not include related data like messages, questions, or summaries (can be added later)
- **Bulk Operations**: No bulk create or delete operations
- **Soft Deletes**: No soft delete functionality (uses database CASCADE deletes)

## 6. Design Considerations

### 6.1 API Design
- **RESTful Conventions**: Follow REST principles with appropriate HTTP methods and status codes
- **JSON Format**: All request/response bodies use JSON with camelCase field names
- **Error Handling**: Consistent error response format (to be defined in middleware or shared error handler)
- **UUIDs**: All IDs are UUIDs for security and scalability

### 6.2 Database Considerations
- **Transactions**: Consider using database transactions for operations that modify multiple tables (not required for this PRD, but good practice)
- **Foreign Key Constraints**: Database schema enforces referential integrity (rubrics → interview plans → interviews)
- **Timestamps**: All timestamps are stored as TIMESTAMPTZ (timezone-aware)
- **Null Handling**: Use `pgtype` types for nullable fields (Text, UUID, Timestamptz)

### 6.3 Handler Structure
- **Consistency**: Follow the existing pattern in `handlers/rubrics.go`:
  - Handler struct with `*db.Queries` field
  - Constructor function `NewXxxHandler(q *db.Queries)`
  - Request/Response structs for JSON marshaling
  - Proper error handling and HTTP status codes
- **Dependency Injection**: Pass `db.Queries` through handler constructors (already established pattern)

### 6.4 Router Integration
- **Route Registration**: Add routes to `internal/api/router.go` in `NewRouter` function
- **Handler Dependencies**: Pass `db.Queries` from `main.go` through `api.Dependencies` struct to handlers
- **Route Naming**: Use kebab-case for URL paths (`/interview-templates`, not `/interviewTemplates`)

### 6.5 Validation
- **Input Validation**: Validate all required fields and data types before database operations
- **Reference Validation**: Verify foreign key references exist before creating related records
- **Enum Validation**: Validate status enums match database ENUM types

### 6.6 Error Messages
- **User-Friendly**: Error messages should be clear but not expose internal implementation details
- **Consistent Format**: Use consistent error response structure (to be standardized)
- **HTTP Status Codes**: Use appropriate status codes (400 for client errors, 404 for not found, 500 for server errors)

### 6.7 Future Considerations
- **Pagination**: Consider adding pagination to list endpoints when data volume grows
- **Caching**: No caching required for MVP, but consider for high-traffic endpoints later
- **Rate Limiting**: Not included in this PRD, but should be considered for production
- **Logging**: Use existing logger infrastructure for request/error logging
- **Testing**: Unit tests and integration tests should be added (separate from this PRD)
