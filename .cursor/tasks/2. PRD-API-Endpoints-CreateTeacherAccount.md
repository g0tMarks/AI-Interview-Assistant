# Product Requirements Document: Create Teacher Account API Endpoint

## 1. Problem Statement

Teachers need a way to create accounts in the AI Interview Assistant system to access the platform and start using its features. Currently, there is no API endpoint for teacher registration, which prevents new users from signing up and accessing the application. Without this endpoint, teachers cannot create accounts to manage rubrics, create interview templates, or conduct interviews with students.

The system needs a secure registration endpoint that allows teachers to create accounts using email and password authentication. This will serve as the foundation for user onboarding, with Single Sign-On (SSO) capabilities to be added in a future iteration.

## 2. Target Users

- **Primary Users**: New teachers who want to create an account to use the AI Interview Assistant platform
- **Secondary Users**: System administrators who may need to understand the registration flow for support purposes

## 3. User Stories

### 3.1 Teacher Registration
- **As a new teacher**, I want to create an account with my email address and password, so I can access the AI Interview Assistant platform.
- **As a new teacher**, I want to provide my full name during registration, so my account is properly identified.
- **As a new teacher**, I want to receive clear error messages if my email is already registered, so I know I need to sign in instead.
- **As a new teacher**, I want my password to be securely stored, so my account information is protected.

## 4. Functional Requirements

### 4.1 POST /teachers/register
**Status**: ⚠️ Needs implementation

**Purpose**: Create a new teacher account with email and password authentication.

**Request Body**:
```json
{
  "email": "string (required)",
  "fullName": "string (required)",
  "password": "string (required)"
}
```

**Response**: 201 Created
```json
{
  "teacherId": "uuid",
  "email": "string",
  "fullName": "string",
  "isEnabled": true,
  "createdAt": "timestamp",
  "updatedAt": "timestamp"
}
```

**Validation**:
- `email` must be a valid email format (RFC 5322 compliant)
- `email` must be unique (not already registered)
- `fullName` must not be empty (trimmed, minimum 1 character)
- `password` must meet security requirements:
  - Minimum 8 characters in length
  - At least one uppercase letter
  - At least one lowercase letter
  - At least one number
  - At least one special character (!@#$%^&*()_+-=[]{}|;:,.<>?)

**Error Responses**:
- 400 Bad Request: Invalid request body, missing required fields, invalid email format, or password does not meet requirements
- 409 Conflict: Email address is already registered
- 500 Internal Server Error: Database error or password hashing failure

**Implementation Notes**:
- Use `CreateTeacher` query from `db.Queries`
- Hash password using bcrypt (or similar secure hashing algorithm) before storing
- Never return `passwordHash` in response
- Check for existing email using `GetTeacherByEmail` before creating account
- Default `isEnabled` to `true` for new accounts
- `teacherId`, `createdAt`, and `updatedAt` are automatically set by database
- Password should be hashed with appropriate cost factor (recommended: bcrypt cost 10-12)

**Security Considerations**:
- Passwords must be hashed using a secure one-way hashing algorithm (bcrypt recommended)
- Never log or expose password values in error messages
- Rate limiting should be considered for production (not in scope for this PRD)
- Email verification could be added in future iteration (not in scope for this PRD)

## 5. Non-Goals

The following features are explicitly **not** included in this PRD:

- **Single Sign-On (SSO)**: OAuth, SAML, or other SSO authentication methods (to be added in future iteration)
- **Email Verification**: No email verification step required before account activation in the current MVP - it will be required later.
- **Password Reset**: No password reset or forgot password functionality (separate feature)
- **Account Activation**: No email-based account activation flow
- **Two-Factor Authentication (2FA)**: No 2FA or multi-factor authentication
- **Account Updates**: No PATCH/PUT endpoints for updating teacher account information (separate feature)
- **Account Deletion**: No DELETE endpoint for teacher accounts (separate feature)
- **Login/Authentication**: No login endpoint or JWT token generation (separate feature)
- **Session Management**: No session creation or management
- **Password Strength Meter**: No real-time password strength feedback in API (frontend may implement)
- **Account Recovery**: No account recovery mechanisms
- **Admin Registration**: No special registration flow for administrators
- **Bulk Registration**: No bulk account creation endpoint
- **Invitation System**: No teacher invitation or referral system

## 6. Design Considerations

### 6.1 API Design
- **RESTful Conventions**: Follow REST principles with appropriate HTTP methods and status codes
- **JSON Format**: All request/response bodies use JSON with camelCase field names
- **Error Handling**: Consistent error response format (to be defined in middleware or shared error handler)
- **UUIDs**: All IDs are UUIDs for security and scalability
- **Security**: Never expose sensitive information (passwords, password hashes) in API responses

### 6.2 Database Considerations
- **Email Uniqueness**: Database schema enforces unique constraint on email field
- **Password Storage**: Password hash is stored as TEXT (nullable to support future SSO-only accounts)
- **Timestamps**: All timestamps are stored as TIMESTAMPTZ (timezone-aware)
- **Default Values**: `isEnabled` defaults to `true`, `created_at` and `updated_at` are set automatically

### 6.3 Password Security
- **Hashing Algorithm**: Use bcrypt (or Argon2) for password hashing
- **Cost Factor**: Use appropriate bcrypt cost (10-12 recommended for balance of security and performance)
- **Password Validation**: Enforce strong password requirements at API level
- **No Plaintext Storage**: Passwords must never be stored in plaintext
- **Hash Comparison**: Use secure comparison functions (constant-time) when verifying passwords (for future login endpoint)

### 6.4 Handler Structure
- **Consistency**: Follow the existing pattern in `handlers/rubrics.go`:
  - Handler struct with `*db.Queries` field
  - Constructor function `NewXxxHandler(q *db.Queries)`
  - Request/Response structs for JSON marshaling
  - Proper error handling and HTTP status codes
- **Dependency Injection**: Pass `db.Queries` through handler constructors (already established pattern)
- **Password Hashing**: Use a password hashing utility or library (e.g., `golang.org/x/crypto/bcrypt`)

### 6.5 Router Integration
- **Route Registration**: Add route to `internal/api/router.go` in `NewRouter` function
- **Handler Dependencies**: Pass `db.Queries` from `main.go` through `api.Dependencies` struct to handlers
- **Route Naming**: Use kebab-case for URL paths (`/teachers/register`, not `/teachersRegister`)

### 6.6 Validation
- **Input Validation**: Validate all required fields and data types before database operations
- **Email Validation**: Use proper email validation (regex or library) to ensure valid format
- **Password Validation**: Enforce password strength requirements before hashing
- **Uniqueness Check**: Verify email is not already registered before attempting to create account
- **Error Messages**: Provide clear, user-friendly error messages without exposing internal details

### 6.7 Error Handling
- **User-Friendly Messages**: Error messages should be clear but not expose internal implementation details
- **Consistent Format**: Use consistent error response structure (to be standardized)
- **HTTP Status Codes**: 
  - 400 for validation errors (invalid format, missing fields, weak password)
  - 409 for conflict (email already exists)
  - 500 for server errors (database failures, hashing errors)
- **Security**: Never expose password hashes, database errors, or internal system details in error responses

### 6.8 Future Considerations
- **SSO Integration**: The database schema already supports nullable `password_hash` to allow SSO-only accounts in the future
- **Email Verification**: Consider adding email verification step in future iteration
- **Rate Limiting**: Should be implemented for production to prevent abuse and brute force attacks
- **Password Reset**: Will need to be implemented as separate feature
- **Account Management**: Update and delete endpoints will be needed in future
- **Logging**: Use existing logger infrastructure for registration events (successful and failed attempts)
- **Testing**: Unit tests and integration tests should be added (separate from this PRD)
- **Audit Trail**: Consider logging registration events for security and compliance
