## Task 4: Create Teacher Handler

**Status**: âś… Complete

**Description**: Create a new handler file for teacher account operations.

**Files to Create**:
- `backend/internal/api/handlers/teachers.go`

**Steps**:
1. Create `TeacherHandler` struct with `q *db.Queries` field
2. Create `NewTeacherHandler(q *db.Queries)` constructor
3. Define `RegisterTeacherRequest` struct:
   - `Email` (string, required)
   - `FullName` (string, required)
   - `Password` (string, required)
4. Define `TeacherResponse` struct (exclude `passwordHash`):
   - `TeacherID` (uuid.UUID)
   - `Email` (string)
   - `FullName` (string)
   - `IsEnabled` (bool)
   - `CreatedAt` (pgtype.Timestamptz)
   - `UpdatedAt` (pgtype.Timestamptz)
5. Implement `RegisterTeacher` method:
   - Decode JSON request body
   - Trim and validate `fullName` (not empty after trimming)
   - Validate email format using email validation utility
   - Validate password strength using password validation utility
   - Check if email already exists using `GetTeacherByEmail`
     - If exists, return 409 Conflict
     - Handle `sql.ErrNoRows` appropriately
   - Hash password using password hashing utility
   - Convert request to `CreateTeacherParams`:
     - Convert hashed password to `pgtype.Text`
   - Call `CreateTeacher` query
   - Convert response from `AppTeacher` to `TeacherResponse`:
     - Convert `pgtype.UUID` to `uuid.UUID`
     - Exclude `passwordHash` from response
   - Return 201 Created with response
   - Handle errors:
     - 400 for validation errors (invalid format, missing fields, weak password)
     - 409 for email already exists
     - 500 for database errors or hashing failures

**Request/Response**:
- Request: POST with JSON body (see PRD section 4.1)
- Response: 201 Created with `TeacherResponse` (no password hash)
- Errors: 400 (validation), 409 (email exists), 500 (server error)

**Acceptance Criteria**:
- Validates all required fields
- Validates email format
- Validates password strength
- Checks email uniqueness before creating account
- Hashes password securely before storing
- Never returns password hash in response
- Properly converts between `pgtype` and standard types
- Returns appropriate HTTP status codes
- Handles all error cases correctly

## Notes

- All handlers should follow the pattern established in `handlers/rubrics.go`
- Use `pgtype` types for database operations and convert to standard types for JSON responses
- Password hashing must use bcrypt with cost factor 10-12
- Never expose password hashes or passwords in API responses or error messages
- Email validation should be RFC 5322 compliant (consider using Go's `net/mail` package)
- Password validation should check all requirements from PRD before hashing
- Check email uniqueness before attempting to create account to avoid unnecessary hashing
- Handle database unique constraint violations gracefully (convert to 409 Conflict)
- All validation should happen before database operations
- Error messages should be user-friendly but not expose internal implementation details
- Consider creating a shared validation package if multiple handlers will need these utilities
